# p01 — Project initialization: Next.js + Prisma habit tracker

## 1) Work summary

- Initialized the HabitLog project in the empty repository: Next.js 16.2.10 (App Router, Turbopack) via `create-next-app`, TypeScript strict mode, Tailwind CSS v4, ESLint 9 (flat config), Prettier, npm.
- Environment repair before scaffolding: the container had no `git` binary (installed via apt) and `/workspace/.git` was a stale 47-byte gitdir pointer to a nonexistent `/.git/modules/benchmark-nextjs-fable`; replaced it with a fresh `git init` on branch `main`. This environment also enforces an npm install-script allowlist — approved `esbuild`, `sharp`, `unrs-resolver`, `prisma`, `@prisma/engines`, `better-sqlite3` (persisted in `package.json#allowScripts`).
- Prisma 7.8.0 set up with SQLite at `./prisma/dev.db`: schema with `Habit` and `CheckIn` models (unique compound index on `habitId + date`), initial migration `20260702045622_init` applied, client generated to `lib/generated/prisma` (gitignored, regenerated on `postinstall`), runtime wired through the `@prisma/adapter-better-sqlite3` driver adapter (required by Prisma 7's Rust-free client).
- Seed script (`prisma/seed.ts`, run with tsx via `prisma db seed`): resets data, creates 3 habits (Morning Run / Read 20 Pages / Meditate — one intentionally without description to exercise the optional field) with random check-ins over the last 14 days (~60% of days, at least 1 each). Seeded run produced 12 / 10 / 8 check-ins.
- Base layout: `TopNav` server component with "HabitLog" logo and Home / Stats / Settings menu, responsive (flex-wrap, breakpoint paddings/gaps); centered `main` container in the root layout; metadata with title template. Placeholder `/stats` and `/settings` pages so the menu has live targets.
- Home page: server component querying non-archived habits via the shared Prisma client, rendered as `HabitCard`s (color swatch + name + optional description) in a 1/2/3-column responsive grid; `force-dynamic` so data is read per-request; empty state points at `npm run db:seed`.
- Directory structure per project rules: `app/`, `components/ui/` (Card), `components/features/` (TopNav, HabitCard), `lib/` (prisma singleton), `actions/` (kept with `.gitkeep` until the first server action).
- README.md (setup + scripts table + structure) and CLAUDE.md (Project Rules + Reporting Rules + Prisma 7 repo notes) written.
- Confirmed dev server runs (HTTP 200, "Ready in 214ms"), and production server renders the 3 seeded habits on `/`.

## 4) Key decisions and reasons

- **Re-initialized git**: `/workspace/.git` was a broken submodule-style pointer to a directory that doesn't exist in this container, and no git binary was installed. Installed git, removed the stale pointer, `git init -b main`, set user identity, and added `/workspace` to `safe.directory` (root vs ubuntu file ownership).
- **Prisma 7 conventions**: Prisma 7 moved CLI config to `prisma.config.ts` and requires a driver adapter for the generated (Rust-free) client — used `@prisma/adapter-better-sqlite3`. Generator output goes to `lib/generated/prisma` (not the `app/` default suggested by `prisma init`) to match the project directory rules; it's gitignored and regenerated by `postinstall`.
- **No `.env` / dotenv dependency**: datasource URL is `process.env.DATABASE_URL ?? "file:./prisma/dev.db"` in `prisma.config.ts` (and mirrored in `lib/prisma.ts` / `prisma/seed.ts`), so a fresh clone works with zero env setup while still allowing overrides. The scaffold's `.gitignore` ignores `.env*`, so a required `.env` would have been a fresh-clone trap.
- **Length constraints in validation layer**: SQLite has no length-limited string types, so name 1–50 and description ≤200 are documented in the schema and will be enforced by server-action validation when mutations arrive (no mutations in this task's scope).
- **`force-dynamic` on the home page**: habit data changes at runtime; also keeps `next build` from needing a database. Static prerendering would freeze seeded data into the build.
- **Habit color via inline style**: `habit.color` is user data from the DB, so it can't be a Tailwind token; everything else uses Tailwind tokens only (zinc scale + theme vars). Noted as an explicit exception in CLAUDE.md.
- **No `'use client'` anywhere yet**: nav is 3 always-visible links (flex-wrap responsive), so no hamburger state was needed; every component is a server component.
- **npm install-script allowlist**: this environment blocks dependency install scripts by default; approved only the 6 toolchain packages that need them (esbuild for tsx, sharp for next/image, prisma engines, better-sqlite3 native build). Approvals persist in `package.json#allowScripts`.
- **Placeholder `/stats` and `/settings` pages**: the required menu should not link to 404s.

If the original history matters, this fresh root commit may need to be grafted/rebased onto it.
- Name/description length validation (1–50 / ≤200) must be added with the first server actions (see Key decisions).
