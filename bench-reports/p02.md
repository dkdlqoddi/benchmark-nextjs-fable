# p02 — Habit CRUD (create / edit / archive / restore / hard delete)

## 1) Work summary

- **Create** — `/habits/new` renders the shared `HabitForm` (name, optional description, 8 preset color swatches as a radio group). Home page gained a "New habit" button.
- **Edit** — `/habits/[id]/edit` is entered from an Edit link on every habit card and reuses the exact same `HabitForm` component, prefilled via `initialValues`; unknown ids return 404 via `notFound()`.
- **Archive** — an Archive button on each home card calls the `archiveHabit` server action (sets `archivedAt`); home only lists `archivedAt: null` habits. `/habits/archived` lists archived habits (with archive date) and offers Restore (`restoreHabit` sets `archivedAt: null`). Home links to the archive ("View archived").
- **Hard delete** — only rendered on the archived list. The Delete button sits in a `ConfirmForm` client wrapper that shows a confirm dialog naming the habit; on confirm, `deleteHabit` removes the habit and its check-ins in one transaction.
- **Server Actions + zod** — all 5 mutations (`createHabit`, `updateHabit`, `archiveHabit`, `restoreHabit`, `deleteHabit`) live in `actions/habits.ts` (`"use server"`); no API routes. `lib/habit-schema.ts` validates server-side with zod 4: name trimmed 1–50 chars, description trimmed ≤200 (empty → null), color must be one of the 8 presets in `lib/habit-colors.ts`.
- **Error/success behavior** — create/update return a `HabitFormState` consumed by `useActionState`: one message per invalid field rendered under that field (with `aria-invalid`/`aria-describedby`), submitted values echoed back so user input survives the failed submit; on success the action calls `revalidatePath("/")` + `redirect("/")`.
- Everything remains a Server Component except two files with a stated reason: `HabitForm` (needs `useActionState`) and `ConfirmForm` (needs the submit-intercepting confirm dialog).
- Verified all flows in a real headless Chromium via Playwright (24/24 checks passed), plus build/typecheck/lint all clean.

## 4) Key decisions and reasons

- **One shared `HabitForm` for create and edit**: the action is injected as a prop (`createHabit` or `updateHabit.bind(null, id)`), so the form neither knows nor branches on mode — the spec's reuse requirement with zero duplication.
- **Validation schema in `lib/`, not `actions/`**: `"use server"` modules may only export async functions, so the zod schema, form-state types, and `parseHabitForm` helper live in `lib/habit-schema.ts` and are imported by the actions.
- **Submitted values echoed back in form state**: React 19 resets uncontrolled forms after an action completes, which would wipe user input on a validation failure; returning `values` in the state and using them as `defaultValue`s preserves what the user typed.
- **No native `required`/`maxLength` attributes**: deliberately omitted so the server-side zod validation (the spec's requirement) is observable and the single source of truth, instead of being masked by browser pre-validation.
- **Explicit check-in deletion in a transaction**: rather than relying on SQLite's FK cascade (whose enforcement depends on the driver's `PRAGMA foreign_keys`), `deleteHabit` runs `checkIn.deleteMany` + `habit.delete` in `prisma.$transaction` — deterministic regardless of pragma state.
- **Native `window.confirm` for the delete dialog**: satisfies "requires a confirm dialog" with the smallest possible client surface (one tiny `ConfirmForm` wrapper) instead of a modal component; the wrapper is generic and reusable in `components/ui/`.
- **Archive/restore/delete don't redirect**: they `revalidatePath` so the current list refreshes in place; only the create/edit forms redirect home per the spec.
- **Playwright installed `--no-save`**: real-browser verification (dialogs, per-field errors, redirects) without adding test tooling to the project's dependencies; the e2e script stays in the session scratchpad, not the repo.
- **Preset colors are data**: swatches render via inline `style` like `habit.color` itself; the CLAUDE.md exception to "Tailwind tokens only" was reworded to cover both.

## 5) Remaining issues/TODOs

- Check-in tracking UI (the actual "check in today" flow) is not part of this task — presumably the next one; `deleteHabit` and the schema are already prepared for it.
- Habit name uniqueness is not enforced (not in the spec); duplicate names are allowed and distinguishable only by description/color.
- The confirm dialog is the native browser one; a styled modal could replace it later without touching the actions.
- In a no-JS browser, forms still work via progressive enhancement, but the delete confirmation (client JS) would not intervene — acceptable given the app requires JS for its enhanced flows anyway.
- `npm audit`: 2 moderate vulnerabilities in transitive dev tooling remain (unchanged from p01).
