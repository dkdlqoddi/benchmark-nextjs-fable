# p03 ‚Äî Daily check-ins + monthly calendar (Asia/Seoul)

## 1) Work summary

- **Home card toggle** ‚Äî every habit card has a "Check in today" button that becomes "‚úì Done today" (filled with the habit's color, `aria-pressed`) once checked; clicking again un-checks. It posts the `toggleTodayCheckIn` server action; the card needs no client JS.
- **Detail page `/habits/[id]`** ‚Äî reached by clicking a habit's name on home. Shows a monthly calendar: weekday header, correctly offset day grid, checked dates filled with the habit's color, today ringed, plus a month check-in count. Prev/next month navigation is plain links (`?month=YYYY-MM`), so the whole page stays server-rendered; malformed month params fall back to the current month.
- **Calendar toggling** ‚Äî each past/today cell is a tiny form whose button submits `toggleCheckInForDate` bound to that date; future cells render `disabled`. The action also re-validates server-side (date-key format + not-in-future) so a forged request can't check a future date. Toggle uses `deleteMany` ‚Üí `create` with a P2002 catch, making double-submits race-safe.
- **Timezone fixed to Asia/Seoul** ‚Äî new `lib/date.ts` centralizes all date logic: `todayKey`/`toDateKey` (Intl `en-CA` with `timeZone: "Asia/Seoul"`), `dateKey`, `addDays`, `isFutureKey`, month helpers (`parseMonth`, `addMonths`, `monthParam`, `monthLabel`, `daysInMonth`, `firstWeekday`). Home, actions, calendar, and the seed script all consume these helpers; nothing computes "today" ad hoc. Correctness holds regardless of the server's TZ (verified below with the server deliberately run at UTC‚àí12).
- CLAUDE.md and README now record the timezone constraint; DB reseeded with Seoul-based keys.
- Everything remains Server Components ‚Äî this task added zero `'use client'` files.

## 4) Key decisions and reasons

- **Intl-based timezone handling, no dependency**: `Intl.DateTimeFormat("en-CA", { timeZone: "Asia/Seoul" })` yields YYYY-MM-DD keys directly; Node 20+ ships full ICU. Everything else (grid offsets, month arithmetic) is pure Gregorian math on `Date.UTC` parts, which no timezone can affect.
- **Server runs verified TZ-independent**: rather than trusting the implementation, the e2e server was started at UTC‚àí12 ‚Äî a timezone whose calendar date differed from Seoul's at test time ‚Äî and the created check-in still landed on Seoul's date.
- **Calendar cells as per-date forms with bound server actions**: keeps the calendar a Server Component (zero client JS added this task), gives progressive enhancement for free, and each click re-renders fresh data via `revalidatePath`.
- **Month navigation via query param links** (`?month=YYYY-MM`) instead of client state: server-rendered, shareable URLs, and `parseMonth` safely falls back to the current Seoul month on malformed input.
- **Defense in depth for future dates**: disabled buttons in the UI *and* re-validation inside the action (`DATE_KEY_PATTERN` + `isFutureKey`), proven by the forced-click test.
- **Race-safe toggle**: `deleteMany` (idempotent un-check) then `create` guarded by a P2002 catch ‚Äî two concurrent submits can't crash or double-insert given the `@@unique([habitId, date])` constraint.
- **Lexicographic date comparison**: YYYY-MM-DD keys order correctly as strings, so "future" checks and month-range queries (`gte`/`lte` on keys) need no Date parsing.
- **Detail page reachable from the card name**: the spec implies browsing to `/habits/[id]`; linking the card title is the least intrusive entry point.
