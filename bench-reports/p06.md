# p06 — Dark mode (user-controllable Light / Dark / Auto)

## 1) Work summary

The app already had complete dark styling (`dark:` variants everywhere, dark chart tokens) but it only followed the OS. "Add dark mode" therefore delivered the missing piece: a user-controllable theme with persistence and no flash of the wrong theme.

- **Class strategy** — `@custom-variant dark (&:where(.dark, .dark *))` in `globals.css` switches every existing `dark:` utility from the `prefers-color-scheme` media query to a `.dark` class on `<html>`; the CSS custom properties (`--background`, `--foreground`, chart tokens) moved from the media query into a `.dark` block, and `color-scheme: light/dark` now makes native controls and scrollbars match.
- **`lib/theme.ts`** — the theme store: `ThemePreference` (`light | dark | system`), localStorage read/write, `.dark` class application, a subscriber set shaped for `useSyncExternalStore` (with cross-tab sync via the `storage` event), and `THEME_INIT_SCRIPT` — a self-contained inline script rendered as the first child of `<body>` that applies the stored theme synchronously before first paint.
- **`components/ui/ThemeToggle.tsx`** — a three-button segmented control (Light / Dark / Auto) in the top nav, `'use client'` with the required justification. The preference is read via `useSyncExternalStore` (null during SSR → nothing highlighted, no hydration mismatch); an effect subscribes to `matchMedia` while in Auto so the page reacts live when the OS preference flips.
- **`app/layout.tsx`** — injects the init script and sets `suppressHydrationWarning` on `<html>` (the script mutates the class pre-hydration by design).
- CLAUDE.md records the standing rule (never reintroduce `prefers-color-scheme` styling; use `dark:`/`.dark`), README documents the toggle.

## 4) Key decisions and reasons

- **Scope read**: with `dark:` styling already complete, the deliverable is choice + persistence + no-FOUC — i.e. the `next-themes` behavior — implemented in ~90 lines instead of adding the dependency, since the app needs exactly one store and one toggle.
- **Class-based variant over media query**: a user override is impossible with pure `prefers-color-scheme`; the Tailwind v4 `@custom-variant` line converts every existing `dark:` class without touching any component.
- **Inline pre-paint script** (first child of `<body>`, `suppressHydrationWarning` on `<html>`): the standard no-flash pattern; localStorage must be read before anything paints, which no React lifecycle can do.
- **`useSyncExternalStore` over `useState`+effect**: the new React hooks lint rule rightly rejects mount-time setState; uSES is the designed tool for client-only external state — SSR renders the "unknown" snapshot (no highlighted option), and cross-tab sync came free via the `storage` event.
- **Three states, not a binary toggle**: removing the ability to follow the OS would be a regression for users who had it; "Auto" keeps the old behavior as an explicit option and remains the default.
- **`color-scheme` property**: native form controls, scrollbars, and the confirm dialog render in the active theme.
