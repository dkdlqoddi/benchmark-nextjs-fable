## 1) Work summary

Added credentials authentication and per-user data isolation to HabitLog:

- **Auth.js v5** (`next-auth@5.0.0-beta.31`, peer-compatible with Next 16) with a **Credentials provider** (email + password). Passwords hashed with **bcrypt** (`bcryptjs` 3.0.3 â€” pure-JS implementation producing standard `$2b$` hashes; cost 10). Config is split per the v5 pattern: `lib/auth-config.ts` (DB-free base: JWT strategy, pages, id-into-session callbacks, secret with a dev fallback preserving the repo's zero-`.env` setup) and `lib/auth.ts` (credentials provider with Prisma + bcrypt, plus the `requireUserId()` helper). `authorize` compares against a constant dummy hash when the email is unknown, so failed logins cost one bcrypt compare either way (no user-enumeration timing signal). No API routes were added: `signIn`/`signOut` are called only inside server actions, keeping project rule 2 intact.
- **Signup (`/signup`), login (`/login`), logout**: server actions in `actions/auth.ts` (zod validation in `lib/auth-schema.ts`; signup auto-signs-in, duplicate email handled via the P2002 race-safe path), shared `AuthForm` client component, session-aware `TopNav` with a Sign out form button.

- **Route protection in middleware**: Next 16 renamed middleware to **proxy** (per the bundled docs: "Starting with Next.js 16, Middleware is now called Proxy"), so the guard lives in `proxy.ts`: without a session every path except `/login`/`/signup` redirects to `/login`; signed-in users are bounced off the auth pages. Cookie-only optimistic check (no DB), as the framework guide prescribes.
- **User scoping on every read and write**: all pages resolve `requireUserId()` and filter by `userId` (detail/edit use scoped `findFirst` â†’ `notFound()`, so a foreign id 404s; `generateMetadata` is scoped too, so a foreign habit's *name* can't leak via the page title). Check-ins are scoped through their habit (`habit: { userId }`).

## 5) Remaining issues

- **JWT sessions are stateless**: `signOut` clears the cookie, but a previously captured token value remains decodable until expiry (verified: old cookie still returned 200 after logout). Revocation would need a DB session strategy or a token denylist â€” out of scope; noted as inherent to the credentials+JWT pattern.
- `AUTH_SECRET` falls back to a hardcoded dev value to preserve the zero-`.env` setup (documented in README and `lib/auth-config.ts`); real deployments must set it.
- No rate limiting / lockout on login attempts, and no password-change or account-deletion flows (not in scope).
- `/settings` remains a placeholder; the account email in `TopNav` is the only account UI.
- `npm audit`: 5 moderate vulnerabilities in transitive dev tooling after the installs (was 2) â€” dev-time only, unchanged policy from p01.
