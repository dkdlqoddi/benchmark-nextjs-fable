# p10 — Target days of the week (dim off-days, target-day streaks)

## 1) Work summary

- **Model**: `Habit.targetDays` — a 7-char `0/1` mask string, index 0 = Sunday … 6 = Saturday (deliberately the same weekday numbering `lib/date.ts` already uses), `@default("1111111")`. Migration `20260702094337_add_target_days` ran **in place** on the populated DB; requirement 4 evidenced by querying right after: all 6 pre-existing habits came out with `targetDays = '1111111'` ("every day"). No reset was needed — the column default is the migration story.
- **Vocabulary**: new `lib/target-days.ts` — `isTargetDaysMask` (7×`[01]`, ≥1 target: the walk helpers rely on that to terminate), `normalizeTargetDays` (guards stale DB data), `isTargetDate`, `latestTargetDayOnOrBefore` / `previousTargetDay` / `nextTargetDay`, and `targetDaysLabel` ("Every day" / "Mon, Wed, Fri"). Added `weekday(key)` to `lib/date.ts` (and refactored `startOfWeek` onto it) since weekday math is date logic.
- **Streak rules** (`lib/streak.ts` rewrite): streaks count **target days only**. `currentStreak` = run of consecutively checked target days ending at the latest target day ≤ today — or at the *previous* target day when that latest one **is** today and still unchecked (today isn't over yet). `longestStreak` links target days via `nextTargetDay`. Off-day "bonus" check-ins neither extend nor break runs; `total` still counts every check-in. With an every-day mask this reduces exactly to the old today-or-yesterday rule, so all original p04 spec cases still pass unchanged.
- **UI**: habit form gained a "Target days" checkbox group (`≥1` enforced by zod via `isTargetDaysMask`; checkboxes fold into the mask server-side in `parseHabitForm`). On non-target days the check-in UI is **dimmed but active**: home card button gets `opacity-60`, label "Check in (off day)" and a title hint; calendar dims off-day headers and cells (unchecked: lighter ink; checked: color fill at `opacity-60`) while keeping them clickable, with a "dimmed days are off-days" caption. Cards, detail header, and stats cards show the target-day label.
- **Seed**: alice's Morning Run → Mon/Wed/Fri, Meditate → weekdays; bob's Gym Session → Tue/Thu/Sat; the rest stay every-day.
- **verify-streak.ts**: rewritten for the new rules — the 9 previous every-day cases (5 spec + 4 extras, unchanged expectations, proving backward compatibility) plus 8 new target-day cases (T1–T8: alive-across-off-day, bonus check-in neutrality, today-target grace, missed-target break, off-day current streak, longest across a missed target, weekend-skipping weekday run). All 17 pass.
- Docs: README (feature paragraph), ARCHITECTURE.md (model, modeling-choices bullet, `lib/` inventory, streak rules, invariant #1).

## 4) Remaining issues

- **Weekly completion rates (`lib/completion.ts`) still treat every day as "possible"** for every habit. Bonus check-ins keep the numerator consistent with the denominator, so the chart remains internally coherent, but a target-day-aware `possible` (per-habit target days elapsed) would be more meaningful — deferred as out of scope (the task asked to change streaks only).
- Off-day dimming on the home card uses `opacity-60` on both checked and unchecked states; a design pass might prefer a distinct off-day treatment for the checked state.
- `normalizeTargetDays` silently falls back to "every day" for an invalid stored mask (defensive against hand-edited DB rows); it does not repair the row.
- Unchanged from p09: no rate limiting on login; JWT sessions not revocable server-side; `npm audit` 5 moderate (dev tooling).
