# p11 — Habit tags + tag-filterable home page

## 1) Work summary

- **Model**: new user-scoped `Tag` (`@@unique([userId, name])`, cascade FK to `User`) with an **implicit many-to-many** to `Habit`. Tag names are stored normalized — trimmed, inner whitespace collapsed, lowercased (`lib/tags.ts`) — and are never shared across users: alice's "health" and bob's "health" are distinct rows. Migration `20260702140653_add_tags` is purely additive (no reset).
- **Input**: the habit form gained a comma-separated "Tags" text field (≤ 5 tags, ≤ 20 chars each, enforced by zod via `parseTagList` transform + refinements; raw string echoed back on error). Kept as plain text input so the form stays server-validated with no new client code.
- **Mutations** (`actions/habits.ts`): `createHabit` attaches tags via user-scoped `connectOrCreate` (`userId_name`); `updateHabit` fully replaces with `tags: { set: [], connectOrCreate }`; both `updateHabit` and `deleteHabit` end with an orphan sweep (`tag.deleteMany({ userId, habits: { none: {} } })`). `deleteHabit` now explicitly detaches tags (`tags: { set: [] }`) inside its transaction — same driver-dependent-FK-cascade caution as check-ins — behind a scoped ownership gate, with a P2025 race guard.
- **Home filter**: `/?tag=name` — the query value is normalized with the same `normalizeTagName`, applied as `tags: { some: { name } }` on top of the existing `userId` scope. A `TagFilter` server component renders "All" + one chip per tag (chips derive from *active* habits only), active chip highlighted with `aria-current`; the summary line reads "Tracking N habits tagged #x", with a dedicated "No habits tagged #x" empty state. Habit cards and the detail header render tag chips linking back to the filtered home page. Everything is server-rendered links — no new client components (still five).
- **Seed**: alice → health/morning/growth/mindfulness across her 3 habits; bob → fitness/mindfulness/evening/health (10 habit–tag links, 8 tag rows).
- Docs updated (README feature note; ARCHITECTURE model block, modeling-choices bullet, `lib/` inventory, routes table).

## 4) Remaining issues

- Filtering is single-tag; multi-tag AND/OR filtering wasn't in scope. The URL scheme (`?tag=`) leaves room for `?tag=a&tag=b` later.
- Tag chips/filter derive from active habits, so a tag used only by archived habits is invisible until restore (kept by the orphan sweep since the habit still references it) — deliberate, but worth knowing.
- If SQLite runs without FK enforcement, deleting a *user* row directly in the DB (outside the app, which never does this) could leave implicit-m2m join garbage; app-level paths detach explicitly.
- Unchanged from p10: weekly completion "possible" ignores target days; login has no rate limiting; JWT sessions not revocable.
