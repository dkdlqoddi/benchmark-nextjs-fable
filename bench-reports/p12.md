# p12 — Search feature (home page ?q=)

## 1) Work summary

Added free-text habit search to the home page, composable with the existing tag filter:

- **`?q=` query param**: matches active habits whose **name, description, or any tag name** contains the query — Prisma `contains` on SQLite `LIKE`, so ASCII case-insensitive. Combined with the `?tag=` filter as an AND (the OR applies only across the three searched fields); both ride on top of the existing `userId` scope.
- **`lib/home-filters.ts`** (new): `normalizeSearchQuery` (trim, collapse whitespace, cap at 50 chars, undefined when empty) and `homeHref` (builds home links preserving tag + search state).
- **`components/features/HabitSearch.tsx`** (new): a **server-rendered plain GET form** (`role="search"`) — search is a read, so no server action and no client JS (the app still has exactly five client components). The active tag rides along as a hidden input; a "Clear" link drops only the query.
- **State-preserving filters**: `TagFilter` chips now carry the current `?q=` (via `homeHref`), the summary line reads "Tracking N habits tagged #x matching “q”", and the filtered empty state names both filters with a "Show all habits" escape link.
- Docs: README feature note; ARCHITECTURE routes row + `lib/` inventory entry.

## 4) Remaining issues

- `contains` on SQLite `LIKE` is case-insensitive for ASCII only — a non-ASCII query (e.g. Korean) matches case-sensitively; acceptable for substring search of this scale. Full-text search (FTS5) would be the upgrade path if habit counts ever warrant it.
- A user typing `%` or `_` in the query hits SQLite LIKE wildcard semantics via Prisma's escaping — Prisma escapes these correctly, so no injection or surprise matches (checked: Prisma parameterizes and escapes `contains`).
- Search covers active habits only (consistent with the page it lives on); the archived page has no search — trivially extendable with the same pattern if wanted.
- Unchanged from p11: single-tag filtering; weekly-completion "possible" ignores target days; no login rate limiting.
