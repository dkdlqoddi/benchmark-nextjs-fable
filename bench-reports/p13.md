# p13 — Test suite (Vitest unit tests + Playwright E2E)

## 1) Work summary

Built the project's test suite from zero:

- **Vitest 4 unit tests** (`tests/unit/`, 89 tests, node environment, `@/` alias via the native
  `resolve.tsconfigPaths` option):
  - `streak.test.ts` (34 tests) — all 17 cases from `scripts/verify-streak.ts` ported verbatim
    (9 every-day + 8 target-day spec cases), plus new edge cases: year boundaries (every-day,
    MWF and Mon-only masks across New Year), leap days (2028, century rules 2000/2100, a
    Tue-only run spanning Feb 29 2028), more target-day combinations (weekend-only mask,
    off-day-only check-ins, broken runs, a 9-day weekday streak bridging a weekend), and
    defensive input handling (invalid/malformed masks fall back to every-day, duplicate dates,
    future-dated runs).
  - `date.test.ts` (24 tests) — timezone boundary cases: the Seoul day flips at exactly
    15:00 UTC (day/year rollover, `todayKey`/`isFutureKey` under a frozen clock), Seoul month vs
    UTC month divergence for `currentMonth`/`parseMonth` fallback, `addDays` across month/year/
    leap boundaries, `weekday`/`startOfWeek` week math, `daysInMonth` century leap rules, month
    arithmetic across year boundaries. The whole suite runs under `TZ=America/New_York` to prove
    the Asia/Seoul fixing is server-TZ independent.
  - `habit-schema.test.ts` (14 tests) — `habitSchema`: 4 valid (full, minimal/trimmed/null
    description, all boundaries at once, tag dedupe) and 7 invalid (empty/51-char name, 201-char
    description, non-preset colors, bad masks, >5 tags, >20-char tag) inputs; `parseHabitForm`
    checkbox→mask folding, per-field error mapping, value echo.
  - `auth-schema.test.ts` (17 tests) — `loginSchema` / `signupSchema`: ≥3 valid and 3 invalid
    each (email normalization, 8/72-char password bounds, 7/73-char rejections); `parseAuthForm`
    never echoes the password back.
- **Playwright E2E tests** (`tests/e2e/`, 3 tests, headless Chromium, single worker):
  - `habit-flow.spec.ts` — the full journey: signup → sign out → login → create habit (with
    description + tag) → check in today (toggle flips to "✓ Done today", `aria-pressed`) → stats
    page shows 1-day current/longest streak and 1 total check-in → calendar detail shows the
    check-in.
  - `user-isolation.spec.ts` — cross-user blocking: Bob's home/stats never show Alice's habit;
    direct navigation to Alice's habit detail and edit URLs returns **404** for Bob; Alice's data
    is untouched afterwards. Plus: signed-out visitors are redirected to `/login`.
  - The suite runs against a **production build** on port 3100 with its own database
    (`prisma/test.db`, reset and re-pushed per run via the `test:e2e:server` script) — the dev
    database is never touched. Accounts use unique per-run emails.
- **Scripts**: `test` (`vitest run`) and `test:e2e` (`playwright test`) added — both headless/CI
  ready (no watch mode, Chromium headless by default). `verify:streak` removed.
- **Absorbed & deleted** `scripts/verify-streak.ts` (its 17 cases live on in `streak.test.ts`);
  README + ARCHITECTURE updated (scripts table, layout, quality gates, `tests/` entries).

## 4) Key decisions and reasons

- **Vitest with `environment: "node"`, no jsdom/Testing Library** — the unit targets are pure
  `lib/` functions and zod schemas; async Server Components can't be unit-tested anyway (per the
  bundled Next.js Vitest guide), so component coverage belongs to the E2E layer.
- **`test.env.TZ = "America/New_York"`** in the Vitest config — the date tests assert Seoul keys
  from explicit UTC instants; running the suite under a non-Seoul, DST-observing TZ makes any
  accidental server-TZ dependence fail loudly.
- **Clock control via `vi.useFakeTimers` + `vi.setSystemTime`** for `todayKey`/`isFutureKey`/
  `currentMonth` so boundary instants (14:59:59Z vs 15:00:00Z) are deterministic.
- **E2E against a production build** (`next build && next start`, per the bundled Playwright
  guide) on **port 3100** with **`DATABASE_URL=file:./prisma/test.db`** — Prisma 7 reads the
  override in both `prisma.config.ts` (CLI `db push`) and `lib/prisma.ts` (runtime), so the dev
  server and `dev.db` stay untouched. The DB file is deleted and re-pushed by `test:e2e:server`
  before each server start.
- **Single Playwright worker + unique per-run emails** — one SQLite writer keeps runs
  deterministic; unique emails make retries and `reuseExistingServer` runs safe without wiping
  state between tests.
- **404-based isolation assertions** — the app's contract is that a foreign habit id behaves
  exactly like a missing one, so the cross-user test asserts response status 404 on both the
  detail and edit routes, plus absence from home/stats, plus Alice's data surviving intact.
- **Semantic selectors** (`getByRole`/`getByLabel`) throughout the E2E helpers — they track the
  accessible UI rather than markup details.
- **`--skip-generate` dropped from `prisma db push`** — the flag no longer exists in Prisma 7
  (client generation is decoupled); caught by the first webServer boot failing.
- Kept `retries: 1` locally (2 in CI): a retry restarts with fresh accounts, absorbing the rare
  case of a run straddling the Asia/Seoul midnight boundary mid-test.

## 5) Remaining issues / TODOs

- `test:e2e` rebuilds the app on every run (~35s of the ~50s total) — correct and CI-realistic,
  but a `PLAYWRIGHT_SKIP_BUILD`-style fast path could be added for local iteration.
- No component-level tests (client islands like `ThemeToggle`/`ConfirmForm` are only covered
  indirectly); jsdom + Testing Library could be layered in later without config changes.
- Server actions are exercised through the browser only; direct unit tests would need Auth.js
  session mocking, which the E2E coverage makes redundant at this scale.
- The check-in E2E asserts an on-target-day check-in; off-day ("bonus") check-in UI and the
  calendar toggle-by-date flow are untested.
- Unchanged from p12: SQLite `contains` is ASCII-case-insensitive only; single-tag filter;
  weekly-completion "possible" ignores target days; no login rate limiting.
