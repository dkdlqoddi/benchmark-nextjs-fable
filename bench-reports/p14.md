# p14 — Investigation: "checked in at 11:55 PM, recorded under the next day"

## 1) Investigation process

1. Traced the check-in **write path**: home card button → `toggleTodayCheckIn(habitId)` →
   `toggleCheckIn(habitId, todayKey())` (`actions/check-ins.ts`); calendar cell →
   `toggleCheckInForDate(habitId, key)` with the cell's rendered date key.
2. Traced the **read/display path**: `CheckIn.date` is a `YYYY-MM-DD` string; home
   (`checkedToday`), stats (streaks), and the calendar all compare stored strings against keys
   from `lib/date.ts` — no `Date` re-parsing anywhere in the render path.
3. Audited for ad-hoc date math: `grep` for `new Date(`, `toLocale*`, `getDay/getMonth/getDate/
   getFullYear` across `app/ components/ actions/ lib/` — the only hit outside `lib/date.ts` is
   `archivedAt: new Date()` (a genuine timestamp, not a calendar key). All day-key computation
   goes through the Asia/Seoul-fixed helpers. No local-timezone parsing bug exists.
4. That leaves the two ways the reported symptom can actually occur (below).

## 2) Root-cause analysis

**Conclusion: partly a real bug (fixed), partly by-design behavior (explained).** The report is
reproducible through two independent mechanisms:

**(A) By design — the user's clock isn't the app's clock.** The calendar is deliberately fixed to
Asia/Seoul (`CLAUDE.md`, `README`, `ARCHITECTURE`): `todayKey()` uses `Intl.DateTimeFormat` with
`timeZone: "Asia/Seoul"` regardless of where the user or server is. For any user west of UTC+9,
11:55 PM local time is already past Seoul midnight (e.g. 23:55 in New York = 12:55 KST the next
day; 23:55 UTC = 08:55 KST the next day), so the check-in is *correctly* stored under the Seoul
date — which reads as "the next day" to that user. Not a code defect; changing it would be a
product decision (per-user timezones), out of scope here.

**(B) A real bug — the home button resolved "today" at execution time, not render time.**
`toggleTodayCheckIn` computed `todayKey()` **when the server action executed**. The button the
user clicks was rendered for a specific day (its label, `checkedToday` state, and off-day dimming
all came from the page's `today`), but the day actually written was whatever "today" was when the
request landed. Around Seoul midnight the two disagree: a tab loaded at 11:5x PM and clicked
minutes later, a slow/retried request submitted at 23:59:5x and processed after 00:00:00 — in
every such case the check-in lands on the **next** day even though the UI said (and the user
believed) they were checking in the shown day. Next morning the mark sits on the wrong calendar
cell — exactly the report. It also corrupts follow-up state: checking in the next evening then
*toggles off* the misplaced record. The calendar cells never had this bug because each cell binds
its explicit date key; the home card was the only mutation with an implicit date.

## 3) Fixes and tests made

**Fix** (mechanism B): the home card now binds the date it was rendered for —
`toggleCheckInForDate.bind(null, habit.id, today)` — so a click always toggles the day the button
displayed, no matter when the request is processed. `today` is the same `todayKey()` value
`app/page.tsx` already used to render the button's state, so intent and effect can't diverge.
The rendered key is never future-dated relative to the server (it came from the server), so the
existing pattern/future-date/ownership guards in `toggleCheckInForDate` all still apply. The now
unused `toggleTodayCheckIn` was deleted.

- `components/features/HabitCard.tsx` — new `today` prop (JSDoc documents the render-time-date
  contract), form binds `toggleCheckInForDate`.
- `app/page.tsx` — passes `today` to `HabitCard`.
- `actions/check-ins.ts` — `toggleTodayCheckIn` removed; JSDoc updated; unused `todayKey` import
  dropped.
- `ARCHITECTURE.md` — actions table + layout comment updated.

**Regression tests** (midnight boundary):

- `tests/unit/check-ins-action.test.ts` (new, 6 tests) — exercises the **real server action**
  against a scratch SQLite database (`prisma/unit-test.db`, schema via `prisma db push` in
  `beforeAll`; `requireUserId`/`revalidatePath` mocked; only `Date` faked so Prisma keeps real
  timers): a toggle processed at 23:59:59 KST stores that day; **the rendered day survives a
  request processed after midnight** (the bug — old wiring stored the next day) and a second
  stale toggle removes that same day; an execution-time `todayKey()` flips between 14:59:59Z and
  15:00:05Z (documents why the report happened); Seoul-tomorrow is rejected just before
  midnight; yesterday becomes toggleable the moment the day flips; malformed keys are rejected.
- `tests/unit/date.test.ts` (+2 tests) — named regression: 23:55 KST stays that Seoul day /
  00:05 KST is the next; and 11:55 PM in New York is already Seoul's next day (mechanism A,
  by design).
- `tests/e2e/habit-flow.spec.ts` — after checking in from the home button, the calendar mark
  must sit on `todayKey()` (computed in the test exactly like the server) and on **no other**
  cell of the month. The suite currently runs in the 15:00–24:00 UTC window where Seoul's date
  differs from UTC's, so this assertion genuinely distinguishes the timezones.
- Supporting: `vitest.config.mts` adds `DATABASE_URL=file:./prisma/unit-test.db` to the test env
  (unit tests can never touch `dev.db`); `.gitignore` adds `/prisma/unit-test.db*`.

## 5) Remaining issues / TODOs

- Mechanism A stands: users outside Asia/Seoul see check-ins under the Seoul date by design. If
  that keeps generating reports, the product-level options are per-user timezone preferences or
  surfacing the app timezone in the UI (e.g. "Today = July 2, Seoul time" near the toggle).
- After the fix, a click from a tab rendered before midnight toggles the *shown* (previous) day;
  the re-rendered page then shows the new day unchecked. Honest, but a toast ("Checked in for
  July 1") would make the behavior clearer.
- The stale-tab scenario is proven at the action layer; a browser-level replay would need
  server-clock control (out of scope for the prod-build E2E setup).
