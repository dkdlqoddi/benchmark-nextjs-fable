# p16 — Refactor: shared helpers for the server-action patterns

## 1) What was extracted, and why

Reviewed all three files in `actions/` (`habits.ts`, `check-ins.ts`, `auth.ts`) plus the two form
modules they lean on, and extracted every pattern that was written out more than once:

- **Ownership check → `lib/ownership.ts` (new)**: `findOwnedHabit(userId, habitId)`. The scoped
  `prisma.habit.findFirst({ where: { id, userId } })` gate appeared verbatim in three places
  (`updateHabit`, `deleteHabit`, `toggleCheckIn`). One named helper now carries the invariant in
  its JSDoc — a foreign id yields `null` exactly like a missing one; never query a habit by bare
  id — so future mutations reach for the helper instead of re-deriving (or forgetting) the
  scoping.
- **Expected-race error guard → `isPrismaErrorCode(error, code)` in `lib/prisma.ts`**: the
  `error instanceof Prisma.PrismaClientKnownRequestError && error.code === "…"` test appeared
  three times (signup P2002 "email taken", check-in toggle P2002 "already created", delete P2025
  "already gone"). The helper lives next to the shared client, and all three action files dropped
  their direct `Prisma` (generated-client namespace) imports.
- **Zod parsing + error response shape → `lib/form-state.ts` (new)**: the two schema modules had
  duplicated both the state *shape* and the mapping *logic*. Now shared:
  - `FormState<Field, Values>` — the `{ status, fieldErrors?, formError?, values? }` shape
    consumed by `useActionState`; `HabitFormState` and `AuthFormState` are now aliases of it
    (structurally identical to their previous hand-written definitions, so no consumer changed).
  - `ParsedForm<Data, State>` — the success/failure union both `parseHabitForm` and
    `parseAuthForm` return.
  - `firstIssuePerField(issues)` — the "at most one message per field, first zod issue wins"
    loop, previously copy-pasted in both parsers.
- **Orphan-tag sweep (within `actions/habits.ts`)**: `deleteOrphanTags(userId)` now returns the
  un-awaited Prisma query, so the same helper serves both the standalone `await` in `updateHabit`
  and the fourth operation of `deleteHabit`'s `$transaction`, which previously duplicated the
  query inline.

**Auth check**: already extracted (`requireUserId()` in `lib/auth.ts`); every data action keeps
calling it as its first line. Deliberately *not* wrapped further: composing it with parsing or
ownership into a single mega-helper would hide control flow (early returns, redirects) behind an
abstraction for zero duplication savings.

**Also deliberately not extracted**: the two-line `revalidatePath(...)` + `redirect("/")` tails.
They vary per action (which paths, whether a redirect follows) and hiding a `redirect()` —
control flow — inside a helper hurts readability more than the two lines cost.

## 2) Changed files

**Added**: `lib/form-state.ts`, `lib/ownership.ts`.

**Modified**:

- `lib/prisma.ts` — `isPrismaErrorCode` added next to the shared client.
- `lib/habit-schema.ts`, `lib/auth-schema.ts` — state/parse types become aliases of the shared
  generics; mapping loops replaced by `firstIssuePerField`; messages, keys, and echo semantics
  untouched (passwords still never round-trip).
- `actions/habits.ts` — `findOwnedHabit` ×2, `isPrismaErrorCode(…, "P2025")`,
  `deleteOrphanTags` reused inside the transaction; `Prisma` import dropped.
- `actions/check-ins.ts` — `findOwnedHabit`, `isPrismaErrorCode(…, "P2002")`; `Prisma` import
  dropped.
- `actions/auth.ts` — `isPrismaErrorCode(…, "P2002")`; `Prisma` import dropped.
- `ARCHITECTURE.md` — §4 `lib/` inventory (new modules added, plus `target-days.ts`/`tags.ts`/
  `home-filters.ts`, which were missing from the list), §7 bullet for `lib/form-state.ts`, §8
  server-actions intro names the shared helpers.

External behavior is identical by construction: same queries, same error codes handled at the
same points with the same rethrow semantics, same state objects (the unit tests assert
`fieldErrors`/`values` with exact `toEqual` shapes), same exported action names and signatures.
