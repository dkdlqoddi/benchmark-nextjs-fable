# p18 — Codebase audit (security / data integrity / performance / a11y) + Critical/High fixes

## 1. Work summary

Audited the entire codebase as a first-time senior reviewer across the four
requested areas and wrote the classified findings to **`AUDIT.md`** (repo root,
kept as a separate deliverable). Result: **0 Critical · 2 High · 7 Medium ·
11 Low**. Runtime claims were verified empirically, not assumed — e.g. probed
`PRAGMA foreign_keys` through the real `@prisma/adapter-better-sqlite3` adapter
(= `1`, dangling insert rejected with `P2003`), confirmed `@auth/core` asserts
its secret per-request, and consulted the bundled Next 16 docs
(`node_modules/next/dist/docs/`) for proxy and server-action security semantics
before touching code.

Both **High** findings were fixed, plus three low-risk Medium/Low items that
fell directly out of the checklist:

- **S-1 (High, fixed):** production silently ran on the committed
  `AUTH_SECRET` fallback → forgeable session JWTs (auth bypass) in any
  deployment that forgot the env var. The fallback is now dev/test-only; in
  production Auth.js fails closed per request (`MissingSecret`). The E2E web
  server (a production build) gets an explicit throwaway secret; README
  documents the requirement.
- **S-2 (High, fixed):** no brute-force protection on the credentials login.
  Added `lib/login-rate-limit.ts` — per-email, in-memory lockout (5
  consecutive failures → 15 min), cleared on success, generic message that
  never discloses account existence, unknown emails throttled identically.
  Covered by 9 new unit tests + a full-stack E2E regression test proving the
  lock blocks even the *correct* password.
- **D-1 (Medium, fixed):** `toggleCheckInForDate` accepted calendar-invalid
  keys ("2026-02-31") arriving via the client-tamperable `.bind()` argument →
  permanent, UI-unremovable junk rows. Added leap-aware `isValidDateKey()` to
  `lib/date.ts` + helper- and action-level tests.
- **D-2 (Medium, fixed):** `updateHabit` 500'd (`P2025`) when the habit was
  deleted between the ownership gate and the update; now returns the same
  "This habit no longer exists." form error the gate produces.
- **A-1a (Medium, fixed) / L-1 (Low, fixed):** metadata labels swapped from
  `text-zinc-400 dark:text-zinc-500` (≈2.6:1 light / ≈3.7:1 dark — below WCAG
  AA) to `text-zinc-500 dark:text-zinc-400` (≈4.7:1 / ≈6.9:1) in 4 files, and
  the archived date now renders via `toDateKey()` (Asia/Seoul) instead of an
  ad-hoc UTC `toISOString()` slice (project-rule violation, off-by-one before
  09:00 KST).

Remaining Medium/Low findings are documented in `AUDIT.md` with locations and
recommendations (see §5). Full suite passes after the fixes: lint, typecheck,
**110/110 unit**, production build, **4/4 E2E**, Prettier check.

## 4. Key decisions and reasons

- **Severity honesty over drama:** nothing qualified as Critical — ownership
  scoping is consistent (verified file-by-file and by the existing E2E
  isolation test), so the worst default-config attack needs either a
  misconfigured deployment (S-1) or a sustained online attack (S-2). Both
  rated High and fixed; AUDIT.md records what was *ruled out* for Critical.
- **S-1 fix shape — fail closed via Auth.js, not a build-time throw:** a
  module-scope throw on missing AUTH_SECRET would break `next build` (which
  runs with `NODE_ENV=production` and evaluates module scope) and the E2E
  server. Instead the fallback is scoped to non-production; `@auth/core`'s own
  per-request `assertConfig` (verified in `lib/utils/assert.js`) then rejects
  every request with `MissingSecret` — deployment misconfig becomes loud and
  safe instead of silent and forgeable. Verified both ways: build passes with
  no secret; E2E production server runs with an explicit test secret.
- **S-2 fix shape — per-email consecutive-failure lockout in the action:**
  this app exposes no `/api/auth/*` routes (no NextAuth handlers are mounted),
  so the login server action is the *only* path to `authorize()` — one
  choke point covers all attack paths. Per-email (not per-IP) because server
  actions don't expose a reliable client IP without proxy headers; per-IP
  belongs in a reverse proxy (documented). In-memory Map is deliberate for a
  single-node SQLite app; the multi-instance limitation is documented in
  JSDoc, AUDIT.md, and README. The lock message never reveals whether the
  account exists, and unknown emails are throttled identically (no
  enumeration signal). Clock is passed in (`now`) to keep the module pure and
  unit-testable.
- **Fixing 3 extra Medium/Low items (D-1, D-2, A-1a/L-1):** all are
  low-blast-radius, checklist-central, and zero-risk to behavior tested by the
  suite; each is marked FIXED in AUDIT.md. Riskier Mediums (D-3 toggle→set
  semantics, A-1b habit-color button contrast, A-2 accessible names) are left
  **documented, not half-fixed**, because they change tested APIs, need a
  design decision, or interact with E2E selectors.
- **E2E regression for S-2 instead of mocking `signIn`:** an action-level unit
  test would need to fake Auth.js; the E2E test exercises the real stack and
  pins the user-visible contract (5 failures → even the correct password is
  refused). `page.reload()` between attempts plus a `waitForResponse` barrier
  makes it deterministic against React 19's post-action form reset.
- **Docs-first per AGENTS.md:** read the bundled Next 16 docs before writing
  code — confirmed `proxy.ts` is the renamed middleware (Node runtime), that
  closures are encrypted but `.bind()` args are not, and that actions must be
  treated as directly POSTable — the latter two justify D-1's server-side
  calendar validation.
