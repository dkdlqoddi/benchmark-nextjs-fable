# p19 — Production-readiness pass: loading/error/404, per-page metadata, bundle-size audit

## 1. Work summary

- **Error handling:** added `app/error.tsx` (route-level boundary below the root
  layout — uses Next 16's `unstable_retry` API, per the bundled docs; the old
  `reset` prop no longer exists) and `app/global-error.tsx` (last-resort
  boundary that replaces the root layout, so it ships its own `<html>/<body>`,
  imports `globals.css`, and re-runs the pre-paint theme script).
- **Custom 404:** `app/not-found.tsx` (title/description metadata, styled like
  the app's empty states, links to home and the archive) — covers both
  unmatched URLs and every `notFound()` call (foreign/missing habit ids). The
  E2E assertion on the default Next 404 text was updated to the custom heading.
- **Loading states:** `loading.tsx` added to `/stats` and `/habits/archived`
  (skeletons mirroring each page, built from a new `components/ui/Skeleton.tsx`
  with an SR-friendly `role="status"` wrapper). Loading files were **deliberately
  NOT placed** at the root or on `/habits/[id]`(+`/edit`) — a boundary above a
  `notFound()` route turns its HTTP 404 into a streamed 200, and the root
  boundary made the home check-in click intermittently vanish (both verified
  empirically; see §5).
- **Metadata:** every page now has a title (inherited template `%s — HabitLog`)
  **and** a description: home, stats, settings, new/edit/archived habit, habit
  detail (`generateMetadata`), login, signup, 404.
- **Bundle sizes:** Next 16 removed the size columns from `next build` output
  (upgrade notes: the metrics were inaccurate for RSC), so sizes were measured
  from the build manifests (shared `rootMainFiles` + each route's
  client-reference chunks, raw and gzip). `/stats` was the outlier — **856 kB
  raw / 244 kB gz**, ~6× the route-specific JS of any other page — caused by
  recharts (+ d3 dependency tree) powering one 8-bar chart. Replaced it with a
  dependency-free, server-rendered HTML/CSS chart (dataviz-skill procedure:
  palette validator run for light+dark, mark specs, hover **and keyboard-focus**
  tooltips, sr-only table kept) and removed `recharts` from `package.json`.
  `/stats` now costs the same as every other route (**513 kB / 146 kB gz**).

## 3. Per-route bundle sizes and analysis

Method: Next 16 no longer prints per-route sizes (`next build` upgrade notes:
"removes the `size` and `First Load JS` metrics … found these to be inaccurate"),
so each route's client JS was computed from the build output itself:
shared `rootMainFiles` (from the per-route `build-manifest.json`) ∪ the route's
client-component chunks (from `page_client-reference-manifest.js`), summing
file sizes under `.next/` (gzip = `gzip -9`).

**Analysis**

- **The outlier and its fix:** `/stats` carried 410 kB raw (116 kB gz) of
  route-specific JS — recharts plus its d3-* dependency tree, hydrating a
  simple 8-bar weekly chart. The replacement renders the same marks as plain
  HTML/CSS on the server: 0 chart JS shipped, tooltips are CSS
  (`:hover`/`:focus-visible`), the accessible table stayed. `/stats`
  route-specific JS fell **84%** (410.4 → 66.9 kB raw; 115.6 → 18.1 kB gz) and
  the route is now byte-identical to the other data pages. The dependency is
  gone from `node_modules` entirely.
- **Cost of the new furniture:** every route gained ~3 kB raw (~1.5 kB gz) —
  the `error.tsx` client boundary plus the loading/not-found chunks. Uniform
  and small; accepted as the price of graceful failure modes.
- **The shared 446 kB baseline** is framework-dominated (React DOM, Next app
  runtime/router). Route-specific JS is now 59–73 kB raw (15–20 kB gz)
  everywhere — forms (`useActionState`) on the form pages, theme/nav bits
  elsewhere. Nothing remains unusually large; further cuts would mean removing
  client interactivity, not trimming fat.

## 5. Key decisions and reasons

- **Loading placement is deliberately partial — this was the task's main
  discovery.** A `loading.tsx` above a route that calls `notFound()` makes the
  response stream, and per the bundled docs (`loading.md`/`not-found.md`
  Status Codes) a streamed not-found returns **HTTP 200** with the 404 UI. The
  E2E suite caught exactly that (`expected 404, received 200` for a foreign
  habit id) — a real correctness regression for anything that trusts status
  codes. Separately, with a root boundary the home page's "Check in today"
  click was intermittently lost right after navigation (page re-rendered
  un-toggled; reproduced in most runs, disappeared completely — 9/9 green —
  once the root boundary was removed). So: skeletons only on segment-local
  boundaries with no `notFound()` beneath (`/stats`, `/habits/archived`);
  home/detail/edit render as before (their queries are single-digit-ms SQLite
  reads — the skeleton bought nothing there anyway).
- **`unstable_retry`, not `reset`:** this Next 16's `error.tsx` receives
  `{ error, unstable_retry }` (verified in `node_modules/next/dist/docs/…/error.md`) —
  writing the classic `reset` prop from training data would have silently
  never re-rendered.
- **`global-error.tsx` replaces the root layout when it fails**, so it renders
  its own document shell, imports `globals.css` itself, re-runs
  `THEME_INIT_SCRIPT` (no white flash for dark-mode users even on the crash
  page), and uses a React `<title>` (metadata exports are unsupported there).
- **Chart rewrite followed the dataviz skill procedure** (read before writing
  chart code): form kept (single-series bars over time), the existing
  `--chart-accent` token re-validated with the six-checks script against both
  surfaces, mark specs applied (≤24 px bars, 4 px rounded data-end, square
  baseline, recessive gridlines), per-bar tooltip on hover **and** keyboard
  focus with full-column hit targets (values lead, labels follow), no legend
  (single series — the card title names it), and the sr-only table retained as
  the AT surface. Keeping the existing chart tokens meant zero changes to
  `globals.css`.
- **Custom 404 kept inside the root layout** (`not-found.tsx`, not the
  experimental `global-not-found.js`): the app has a single root layout, so
  the standard convention composes with TopNav and the theme for free.
- **E2E 404 assertions were kept on status codes**, not just UI text — they now
  double as a regression guard for the streamed-200 pitfall.
