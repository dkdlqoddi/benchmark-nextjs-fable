# p22 — Retrospective: the 3 weakest parts, with improvement plans (REVIEW.md)

## 1. Work summary

Wrote **`REVIEW.md`** (repo root, separate deliverable): a critical
retrospective picking the three weakest / most regrettable parts of HabitLog,
each with an evidence-based justification and a staged improvement plan.
Selection criteria stated up front — user impact × coupling depth ×
avoidability at decision time — with already-fixed items (p18 security, p19
bundle) excluded. **No code was modified** (verified: the working tree
contains only the two new markdown files).

The three picks:

1. **The daily check-in is a blind, toggle-semantics round trip** — the app's
   hottest interaction has no pending state and no optimistic update
   (`useOptimistic`/`useFormStatus` appear nowhere), and toggle semantics make
   a double-submit silently invert intent (audit D-3). The p19 loading-
   boundary incident (clicks intermittently lost in E2E) showed how fragile
   the flow is to timing. Plan: desired-state action
   (`setCheckInForDate(habitId, date, checked)` — idempotent by construction)
   plus one small `CheckInButton` client island; port the p13/p14
   midnight-boundary tests intent-for-intent; ~half a day.
2. **The calendar is hardwired to Asia/Seoul for every user** — the policy
   behind the project's one real user incident (p14, "11:55 PM check-in
   recorded next day": the stale-tab race was fixed, the policy half remains),
   silent in the UI, and load-bearing through `todayKey()` everywhere dates
   exist. Plan: (interim) surface "Today = …, Seoul time" in the UI, then
   `User.timezone` with IANA validation, compiler-enforced threading of a
   `tz` parameter through `lib/date.ts`, a Settings picker (finally
   un-placeholdering that page), and a mirrored non-Seoul test suite; 1–2
   days, staged behind the product decision.
3. **Habit colors are raw hex strings stored as row data** — presentation
   frozen into user rows (any palette change is a data migration), one hex
   serving both light and dark surfaces, and the measured consequence that
   white-on-color fails WCAG AA on all 8 presets (audit A-1b, 2.1–4.2:1).
   Plan: keyed palette (`colorKey` + per-mode fill/ink validated with the
   dataviz six-checks script), backfill migration over the 8-value whitelist,
   render through the palette map — closing A-1b/A-1c structurally; ~a day.

Honorable mentions recorded so the cut line is visible: the in-memory login
rate limiter (own p18 choice — a durable SQLite table was one migration
away), irrevocable JWT sessions (framework-constrained), the non-atomic
orphan-tag GC (D-4), and ARCHITECTURE.md's missing update discipline.

## 4. Key decisions and reasons

- **Picked regrets, not just defects:** each item is a *decision* that aged
  badly (toggle semantics, TZ pinning, hex-in-DB), evidenced by incidents or
  measurements from this project's own history (p14 user report, p19 E2E
  click loss, p18 contrast math) — not a re-listing of the p21 TODO
  inventory, though overlapping audit IDs (D-3, A-1b/c) are cross-referenced.
- **Every plan is staged with a cheap first step** (idempotent action, a
  UI timezone hint, a keyed palette constant) so any of them can start
  without committing to the full change.
- **Included a self-authored choice in the honorable mentions** (the p18
  in-memory limiter) — a retrospective that only criticizes inherited code
  isn't one.
- **No code changes**, per the task — even trivially tempting ones (the
  timezone hint) were left as plan items.
